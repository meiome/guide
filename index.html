Scegli guida

Generatore commit

Scrivi il messaggio, copia il comando completo: aggiunge tutti i cambiamenti e fa il commit in un colpo solo.

git add -A && git commit -m ''

Il messaggio viene racchiuso in apici singoli: la shell non interpreta $, `, ! né altri metacaratteri. Gli apostrofi dentro al testo vengono gestiti automaticamente.

Creare o clonare un repository

Per creare un nuovo repository locale nella cartella corrente:

git init

Per clonarne uno esistente:

# da un server remoto (HTTPS/SSH)
git clone https://github.com/utente/repo.git

# da una cartella locale
git clone /percorso/del/repository

I tre "alberi" di git

Il repository locale è composto da tre aree:

  • Directory di lavoro — i file come li vedi nella cartella.
  • Index (staging) — l'area dove "prepari" ciò che andrà nel prossimo commit.
  • HEAD — il riferimento all'ultimo commit fatto.

Il flusso base è: modifichi → git add (sposta nell'index) → git commit (sposta in HEAD) → git push (sincronizza col remoto).

Aggiungere & fare commit

git add nomedelfile       # un file specifico
git add .                 # tutto dalla cartella corrente in giù
git add -A                # tutto il repo (più comune)
git commit -m "messaggio"

Differenze tra -u, . e -A

Comando Modificati Nuovi Eliminati Ambito
git add -uSìNoSìTutto il repo
git add .SìSìSìDalla cartella corrente in giù
git add -ASìSìSìTutto il repo

Push verso il remoto

Invia i commit locali al server:

git push origin main

Se il repository locale non è ancora collegato a un remoto:

git remote add origin https://github.com/utente/repo.git
git push -u origin main

Il flag -u imposta il tracking: dopo la prima volta basta git push.

Branch

I branch servono a sviluppare feature isolate senza toccare main.

git checkout -b feature_x   # crea e passa al nuovo branch
git checkout main           # torna al main
git branch -d feature_x     # elimina il branch locale
git push origin feature_x   # rende il branch visibile sul remoto

Aggiornare & unire

Per sincronizzare la directory di lavoro con il remoto:

git pull

Pull sicuro (senza merge inattesi)

Un git pull "normale" può generare automaticamente un merge commit se il branch locale e quello remoto sono divergenti. Spesso non è quello che vuoi. Il flag --ff-only (fast-forward only) dice a git: aggiorna solo se puoi farlo spostando in avanti il puntatore, senza creare commit di merge. Se non è possibile, git si ferma e ti avvisa.

git pull --ff-only

Pull su una cartella diversa da quella corrente

Il flag -C <cartella> esegue il comando git come se fossi in quella cartella, senza doverci fare cd. È comodo in script, alias, o quando vuoi aggiornare più repository di fila.

git -C /percorso/del/repo pull --ff-only

Esempio pratico — aggiornare due progetti in sequenza da qualsiasi posizione:

git -C ~/progetti/sito pull --ff-only
git -C ~/progetti/api pull --ff-only

Se il pull non può essere fast-forward (hai commit locali divergenti), il comando esce con errore e il repo resta invariato — così non ti ritrovi con merge automatici indesiderati.

Merge & diff

Per unire un altro branch nel tuo branch attuale:

git merge altro_branch

Per vedere le differenze prima di fare merge:

git diff branch_sorgente branch_target

In caso di conflitti, git segnala i file: apri, risolvi, poi git add + git commit.

Tag

I tag marcano versioni/rilasci. Servono 7+ caratteri dell'hash del commit:

git log                   # trova l'hash del commit
git tag 1.0.0 1b2e1d63ff

Annullare modifiche locali

Scartare le modifiche a un file non ancora committato:

git checkout -- nomedelfile

Buttare via tutto quello che hai in locale e tornare allo stato del remoto:

git fetch origin
git reset --hard origin/main
Attenzione: reset --hard è irreversibile: perdi tutte le modifiche locali non pushate.

Comandi utili

git status                        # stato della working directory
git log --oneline --graph --all   # cronologia visuale
git config --global color.ui true # output colorato
git config --global format.pretty oneline
gitk                              # interfaccia grafica integrata

Forzare la rigenerazione dei container

Quando modifichi il Dockerfile, il docker-compose.yml, o vuoi partire "pulito" ignorando le cache:

Solo ricreare i container (senza rebuild immagine)

docker compose up -d --force-recreate

Usa se hai cambiato variabili d'ambiente o parametri del compose ma non il Dockerfile.

Rebuild immagine + ricreare i container

docker compose up -d --build --force-recreate

Uso più comune quando cambi il Dockerfile o il codice copiato nell'immagine.

Rebuild completo senza cache

docker compose build --no-cache && docker compose up -d --force-recreate

Rifà ogni layer da zero. Utile se sospetti che la cache contenga versioni vecchie (es. dopo un apt-get che non aggiorna).

Teardown totale e partenza pulita

docker compose down -v && docker compose up -d --build
Attenzione: il flag -v elimina anche i volumi nominati → perdi i dati (database, upload, ecc.). Ometti -v se vuoi conservarli.

Equivalente senza compose

docker rm -f NOME && docker build -t NOME . && docker run -d --name NOME NOME
